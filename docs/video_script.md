# תסריט וידאו - מטלה 6 בהנדסת אבטחה

## 1. פתיחה

שלום, בסרטון הזה אדגים מערכת התחברות פגיעה שנכתבה בשפת C.

המערכת תומכת ב:

* הרשמת משתמשים
* התחברות רגילה
* שמירת שמות משתמשים וסיסמאות בקובץ
* התקפת גלישת משתנה
* הדגמת דריסת כתובת חזרה
* הדגמת קריאה לפונקציה מוסתרת
* התקפת תזמון

הפרויקט פגיע בכוונה ונוצר רק לצורך המטלה האקדמית הזו.

---

## 2. מבנה הפרויקט

קודם אציג את מבנה הפרויקט.

התיקיות המרכזיות הן:

* src: מכילה את קוד המקור בשפת C.
* data: מכילה את קובץ המשתמשים.
* payloads: מכילה payloads והערות על ההתקפות.
* docs: מכילה תיעוד.
* bin: מכילה את הקבצים המקומפלים.

קבצי ה־C המרכזיים הם:

* server.c
* timing_attack.c

---

## 3. קומפילציה

עכשיו אקמפל את השרת הפגיע.

פקודה:

gcc src/server.c -o bin/server -g -fno-stack-protector -no-pie

הדגל g- מוסיף מידע דיבאג, שנדרש עבור הדיבאגר.

הדגל fno-stack-protector- מבטל את הגנת המחסנית לצורך ההדגמה במטלה.

הדגל no-pie- משאיר את כתובות הפונקציות קלות יותר לבדיקה בזמן דיבאג.

עכשיו אקמפל את תוכנית התקפת התזמון.

פקודה:

gcc src/timing_attack.c -o bin/timing_attack -g

---

## 4. הדגמת התחברות רגילה

עכשיו אריץ את השרת.

פקודה:

./bin/server

התוכנית מדפיסה את הכתובות של פונקציות חשובות:

* authenticated_area
* dump_users

עכשיו אבחר באפשרות 2 עבור התחברות רגילה.

אכניס:

שם משתמש:
admin

סיסמה:
secret

ההתחברות מצליחה והתוכנית נכנסת לאזור המוגן.

עכשיו אנסה סיסמה שגויה.

שם משתמש:
admin

סיסמה:
wrong

ההתחברות נכשלת.

זה מציג את ההתנהגות הרגילה של המערכת.

---

## 5. התקפת גלישת משתנה

עכשיו אדגים את ההתקפה הראשונה: גלישת משתנה.

אבחר באפשרות 3.

קודם אכניס קלט קצר רגיל:

abc

הפלט מציג:

Before input: authenticated = 0
After input: authenticated = 0
Access denied.

עכשיו אכניס את ה־payload:

AAAAAAAAAAAAAAAAA

ה־payload הזה מכיל 17 תווים של A.

ה־buffer של הסיסמה בתוך ה־struct הוא בגודל 16 בתים.
המשתנה authenticated נמצא אחרי ה־buffer של הסיסמה.

16 התווים הראשונים של A ממלאים את ה־buffer של הסיסמה.
התו ה־17 גולש אל המשתנה authenticated.

ערך ה־ASCII של A הוא 65, ולכן authenticated הופך להיות 65.

הפלט מציג:

Before input: authenticated = 0
After input: authenticated = 65
Authentication passed.

כלומר, התוכנית עקפה את האימות בלי לדעת את הסיסמה האמיתית.

---

## 6. דיבאגר עבור גלישת משתנה

עכשיו אפתח את התוכנית בדיבאגר.

פקודה:

gdb ./bin/server

אשים breakpoint בתוך הפונקציה variable_overflow_login.

פקודה:

break variable_overflow_login

לאחר מכן אריץ את התוכנית.

פקודה:

run

אבחר באפשרות 3 כדי להגיע לפונקציית גלישת המשתנה.

עכשיו הדיבאגר עוצר בנקודת העצירה.

אבצע disassemble לפונקציה.

פקודה:

disassemble variable_overflow_login

כאן אפשר לראות את הוראות המכונה של הפונקציה לאחר הקומפילציה.

אפשר גם לבדוק את הכתובות של שדות ה־struct.

פקודות:

print &data
print &data.password
print &data.authenticated

זה מראה שה־buffer של הסיסמה והמשתנה authenticated קרובים בזיכרון.

הגלישה מתרחשת בגלל שהתוכנית משתמשת ב־strcpy בלי לבדוק את גודל הקלט.

---

## 7. הדגמת דריסת כתובת חזרה

עכשיו אדגים את הרעיון של ההתקפה השנייה: דריסת כתובת חזרה.

אבחר באפשרות 4.

התוכנית מדפיסה את הכתובות של פונקציות היעד:

* authenticated_area
* dump_users

הפונקציה הפגיעה היא return_address_vulnerable.

היא מכילה:

char buffer[64];

לאחר מכן היא מעתיקה את הקלט של המשתמש לתוך ה־buffer באמצעות strcpy.

הבעיה היא ש־strcpy לא בודקת את אורך הקלט.

קלט ארוך יכול לדרוס מידע שנמצא אחרי ה־buffer על המחסנית, כולל כתובת החזרה השמורה.

רעיון ה־payload הוא:

בתים למילוי + כתובת פונקציית יעד

הבתים למילוי מגיעים עד כתובת החזרה השמורה.
כתובת פונקציית היעד מחליפה אותה.

כך אפשר להסיט את זרימת התוכנית אל authenticated_area, או אל dump_users.

---

## 8. דיבאגר עבור דריסת כתובת חזרה

עכשיו אראה את החלק הזה בדיבאגר.

פקודה:

gdb ./bin/server

אשים breakpoint:

break return_address_vulnerable

אריץ את התוכנית:

run

אבחר באפשרות 4.

כאשר הדיבאגר עוצר, אבצע disassemble לפונקציה:

disassemble return_address_vulnerable

אציג גם את כתובות הפונקציות החשובות:

print authenticated_area
print dump_users

המטרה כאן היא להראות איך מתכננים את ה־payload:

1. למצוא את ה־buffer על המחסנית.
2. למצוא את המרחק עד כתובת החזרה השמורה.
3. להחליף את כתובת החזרה השמורה בכתובת של פונקציית יעד.

זה מדגים איך גלישת buffer במחסנית יכולה לשנות את זרימת התוכנית.

---

## 9. קריאה לפונקציה המוסתרת dump_users

בתוכנית קיימת פונקציה מוסתרת בשם dump_users.

הפונקציה פותחת את הקובץ:

data/users.txt

ומדפיסה את כל שמות המשתמשים והסיסמאות.

הפונקציה לא מוצגת כאפשרות רגילה בתפריט.
עם זאת, היא עדיין קיימת בזיכרון.

אם תוקף מצליח להסיט את זרימת התוכנית אל dump_users,
התוכנית תדפיס את קובץ המשתמשים.

זה מדגים למה פונקציות מוסתרות עדיין מסוכנות כאשר קיימת חולשת זיכרון.

---

## 10. הדגמה ידנית של דליפת תזמון

עכשיו אדגים ידנית את דליפת התזמון.

פקודה:

./bin/server timing admin s

דוגמת פלט:

result=0
time=84039

עכשיו אנסה תחילית נכונה ארוכה יותר:

./bin/server timing admin se

דוגמת פלט:

result=0
time=176824

הניחוש השני לוקח יותר זמן כי גם s וגם e נכונים.

כלומר, זמן התגובה מדליף מידע על הסיסמה.

---

## 11. תוכנית התקפת התזמון

עכשיו אריץ את תוכנית התקפת התזמון.

פקודה:

./bin/timing_attack

התוכנית מבקשת שם משתמש.

אכניס:

admin

תוכנית ההתקפה מנסה תווים אפשריים אחד אחרי השני.
עבור כל ניחוש, היא מודדת את זמן התגובה.

התו עם זמן התגובה הארוך ביותר הוא כנראה התו הנכון הבא.

התוכנית ממשיכה עד שהיא מוצאת את הסיסמה המלאה.

תוצאה צפויה:

Password found: secret

זה מדגים איך אפשר לשחזר סיסמה באמצעות מדידות זמן.

## 12. סיכום

בסרטון הזה הדגמתי:

1. מערכת התחברות רגילה בשפת C.
2. עקיפת אימות באמצעות גלישת משתנה.
3. רעיון של דריסת כתובת חזרה בעזרת דיבאגר.
4. קריאה לפונקציה מוסתרת בשם dump_users באמצעות שינוי זרימת התוכנית.
5. התקפת תזמון שמבוססת על השוואת סיסמאות לא בטוחה.

הלקחים המרכזיים מבחינת אבטחה הם:

* לא להשתמש בפונקציות העתקה לא בטוחות בלי בדיקת גודל.
* לא לשמור סיסמאות כטקסט גלוי.
* לא להסתמך על פונקציות מוסתרות כאמצעי אבטחה.
* להשתמש בהשוואת סיסמאות בזמן קבוע.
* לקמפל תוכנות אמיתיות עם הגנות זיכרון פעילות.

הפרויקט הזה פגיע בכוונה ומשמש לצורכי למידה בלבד.
